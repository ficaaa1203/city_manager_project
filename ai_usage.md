# AI Usage Documentation – Phase 1

## Tool Used
Claude (claude.ai) – used during implementation of the filter command.

## What I Asked
I provided the following Report struct definition and asked for two functions:

1. `parse_condition(const char *input, char *field, char *op, char *value)`
   – splits a condition string like `severity:>=:2` into its three parts

2. `match_condition(Report *r, const char *field, const char *op, const char *value)`
   – returns 1 if a report record matches the condition, 0 otherwise

## What Was Generated
Both functions were generated by the AI. `parse_condition` uses `strchr` to find
the two `:` separators and copies each part into the output buffers.
`match_condition` uses a series of `strcmp` checks on the field name, converts
the value to the right C type (int for severity, string for category/inspector,
time_t for timestamp), then applies the operator.

## What I Changed
- Added null terminator safety (`field[63] = '\0'` etc.) to prevent buffer overflows
- Verified that `atoi` is used correctly for severity and `atol` for timestamp
- The AI did not handle the case where an unsupported operator is passed for
  string fields (e.g. `category:>:road` makes no sense), I noted this limitation
  but left it as-is since the spec does not require it

## What I Learned
- How `strchr` can be used to tokenize strings without `strtok`
- That type conversion before comparison is critical, comparing an int field
  with a raw string would give wrong results
- The AI generated correct logic but I had to review each line carefully,
  especially the pointer arithmetic in `parse_condition`
